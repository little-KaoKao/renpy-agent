## 1. 分支 `feat/workspace-unification` ★ P0 根节点

### 1.1 背景

本仓库目前有两套互不相通的 workspace IO:

- **v0.2 串行 pipeline**:三份聚合 JSON 落在
  `runtime/games/<story>/workspace/planner.json` / `writer.json` / `storyboarder.json`,
  由 [src/pipeline/workspace.ts](../../../src/pipeline/workspace.ts) 管。
- **v0.6 V5 Planner/Executer**:per-URI `workspace://<kind>[/<slug>]`,落在
  `workspace/project.json` / `workspace/characters/<slug>.json` / `workspace/scenes/<slug>.json` 等,
  由 [src/agents/workspace-io.ts](../../../src/agents/workspace-io.ts) + [src/agents/workspace-index.ts](../../../src/agents/workspace-index.ts) 管。

直接后果:用户跑 `renpy-agent v5 ...` 生成的项目,**不能** 被 `renpy-agent modify` 读写——
[src/pipeline/modify.ts](../../../src/pipeline/modify.ts) 只认 v0.2 聚合 JSON。这是 V5 推广的最大阻塞。

### 1.2 目的

让两套调度(v0.2 / V5)共享同一份 workspace,使得无论哪个入口生成项目,
`modify character` / `modify dialogue` / `reorder shots` / `rebuild` 都能工作。

per-URI 为**主**,聚合 JSON 为**兼容层**(`--resume` 快路径 + v0.2 老产物兼容)。

### 1.3 分支信息

- Base:`main`(commit 12eb60a)
- 分支名:`feat/workspace-unification`
- 预期 PR 大小:中等(~500 行变更 + 测试)
- 关键路径:**是**(tier2-tools / modify-chain 都等它)

### 1.4 允许碰的文件

- [src/pipeline/workspace.ts](../../../src/pipeline/workspace.ts) ← 新增 `saveStoryWorkspacePerUri`
- [src/pipeline/run-pipeline.ts](../../../src/pipeline/run-pipeline.ts) ← 追加 per-URI 双写
- [src/pipeline/modify.ts](../../../src/pipeline/modify.ts) ← 改读 per-URI
- [src/pipeline/rebuild.ts](../../../src/pipeline/rebuild.ts) ← 同步兼容

- 对应 `.test.ts`

### 1.5 绝对不要碰(互斥)

### 1.6 具体做法

1. **扩 `WorkspaceKind` 联合类型**(如需)
   当前只支持 6 种(`project / chapter / character / scene / script / storyboard`)。
   本分支目标是让 modify 链条通,**不**扩新 kind——留给 `feat/v5-tier2-tools` 分支
   (它需要新增 `cutscene` / `bgmTrack` / `voiceLine` / `sfx` / `uiDesign` / `prop` /
   `bugReport`)。
2. **`src/pipeline/workspace.ts` 新增 `saveStoryWorkspacePerUri(gameDir, snapshot)`**
   把 planner / writer / storyboarder 三份结构拆成 V5 per-URI 格式:

   - `planner.projectTitle / genre / tone` → `workspace://project`
   - `planner.chapterOutline` → `workspace://chapter`(单例)
   - `planner.characters[i]` → `workspace://character/<slug>`
   - `planner.scenes[i]` → `workspace://scene/<slug>`
   - `writer` → `workspace://script`(单例)
   - `storyboarder` → `workspace://storyboard`(单例)

   参考 `writeWorkspaceDoc` 的用法。slug 用 `slugForFilename`(步骤 4 后的增强版)。
3. **`run-pipeline.ts` 在现有 `saveStoryWorkspace()` 调用处追加 `saveStoryWorkspacePerUri()`**
   保留聚合 JSON 不删(`--resume` 快路径 + v0.2 老产物兼容)。加一行注释:
   "聚合 JSON 为兼容层,v0.8 起只作为 --resume cache"。
4. **CJK slug 健壮化**(解决"白樱"/"樱白"撞 slug 的问题)
   [src/assets/download.ts:61-67](../../../src/assets/download.ts#L61) 的 `slugForFilename`
   对非 ASCII 名 strip 后 fallback `'asset'`,多中文角色撞 slug。
   改为:ASCII 结果为空时,fallback 到 `char_<sha1_8>`(SHA-1 前 8 位 hex)。
   同步改:

   - [src/agents/workspace-index.ts:27](../../../src/agents/workspace-index.ts#L27) URI 正则,
     接受下划线/连字符/8-hex
   - [src/assets/logical-key.ts](../../../src/assets/logical-key.ts) 所有 `logicalKeyFor*`
     继续走 `slugForFilename`(天然受益)
5. **重写 `modify.ts` 三个 helper 为读 per-URI**

   - `modifyCharacterAppearance`:
     - 读 `workspace://character/<slug>`(slug 由 `slugForFilename(characterName)` 算)
     - 找不到时 fallback 读聚合 JSON 兼容老产物
     - 改完 per-URI **同时**更新聚合 JSON(若存在),两份保持一致
     - AssetRegistry 的回落逻辑保留(status `ready` → `placeholder`)
   - `modifyDialogueLine`:读 `workspace://storyboard`,改 shot 的 `dialogueLines`,双写
   - `reorderShots`:读 `workspace://storyboard`,重排 + 重编号,双写
6. **`rebuild.ts` 同步调整**
   从 per-URI 组装 `PlannerOutput` / `WriterOutput` / `StoryboarderOutput` 内存结构,
   调 Coder + QA。优先读 per-URI,缺失字段回落聚合 JSON。
7. **`run-v5.ts` 暂不接 Coder/audio-ui**
   本分支**不**修改 V5 跑完后的动作(v0.7 里 `feat/v5-tier2-tools` 或更晚的分支补)。
   本分支目标只是"modify 能读 V5 产物",不是"V5 能产出完整 Stage A"。

## 3. 分支 `feat/registry-concurrency`

### 3.1 背景

[src/assets/registry.ts](../../../src/assets/registry.ts) 的 `AssetRegistry` 是单一 JSON 文件
(`asset-registry.json`),`loadRegistry → upsertEntry → saveRegistry` **不是原子**。
所以 [src/pipeline/audio-ui.ts:99-110](../../../src/pipeline/audio-ui.ts#L99) 被迫三组串行
(BGM → Voice → SFX),注释明说:

> The three registry-writing batches must run sequentially to avoid racing on
> `asset-registry.json`.

一部 50 行对白的项目:5 BGM + 30 voice + 10 SFX 串行 ~8-12 分钟;
visual stage 同样问题,角色和场景批次串行。并发能把资产生成从 10 分钟降到 2 分钟。

### 3.2 目的

让 registry 支持并发写,释放 audio-ui / visual stage 的并发度。

### 3.3 分支信息

- Base:`main`
- 分支名:`feat/registry-concurrency`
- PR 大小:中等
- 与 `feat/workspace-unification` 无冲突(走不同 IO),可完全并行

### 3.4 允许碰的文件

- [src/assets/registry.ts](../../../src/assets/registry.ts)
- [src/assets/swap.ts](../../../src/assets/swap.ts)
- [src/pipeline/audio-ui.ts](../../../src/pipeline/audio-ui.ts)
- [src/pipeline/visual-stage.ts](../../../src/pipeline/visual-stage.ts)
- 对应测试 + 新增并发压力测试
- 可能需要新增 `src/assets/registry-lock.ts`(方案 A)

### 3.5 绝对不要碰

- [src/pipeline/workspace.ts](../../../src/pipeline/workspace.ts) / [modify.ts](../../../src/pipeline/modify.ts) ← `feat/workspace-unification`
- `src/executers/*/generate-*.ts` ← **函数签名不要改**(不然和后续 tier2 分支冲突)
- `src/executers/*/tools.ts` ← tier2 分支

### 3.6 方案选型(分支 AI 自己评估,在 PR 写理由)

**方案 A(推荐,改动最小):per-entry 文件**

- 把 `asset-registry.json` 拆成 `asset-registry/<placeholderId>.json` 每条一个文件
- `loadRegistry` 变成扫目录,`upsertEntry` 直接写单个文件——天然并发安全
- 保留 `asset-registry.json` 作为索引(缓存 + 向后兼容),每次 `saveRegistry` 全量重写一次
- 读路径优先读目录,索引失效时自动重建
- placeholderId 里可能有 `:` 等字符,文件名需要 URL-encode

**方案 B:加文件锁**

- 用 `proper-lockfile` 或等价库包住 `loadRegistry` / `upsertEntry` / `saveRegistry` 三连
- 结构不变,最小侵入,但引入新依赖

- 够用当前场景(单进程单 CLI),不解决跨进程并发
- 最简单,但如果 v0.8 出现并发 CLI 可能又要改

**推荐 A**。PR 必须给出:

- 选了哪个 + 理由
- 迁移策略(老项目启动时怎么把单文件转成目录)
- 如果选 A,索引文件(`asset-registry.json`)是否仍保留、何时重建

### 3.7 除方案外的通用改动

## 5. 分支 `perf/prompt-cache`

### 5.1 背景

Planner 的 system prompt 里塞了整份 [src/schema/galgame-workspace.ts](../../../src/schema/galgame-workspace.ts)
(~345 行 TS 作为 Planner 的 schema 宣讲),V5 Executer 的 7-rule prompt + POC 描述每次
`handoff_to_agent` 都重建。Claude API 的 prompt cache 目前**一行没用**——
[src/llm/claude-client.ts](../../../src/llm/claude-client.ts) 里没有任何 `cache_control`。

一次 V5 完整跑(7+ tasks)如果开 cache:

- Planner system prompt 每次 ~3000 token,7 次 cache 命中 → 省 ~18k input token
- Executer 7-rule prompt ~1500 token,每个 task 3-5 turn,累计可省 ~30k input token
- 按 Sonnet 4.6 $3/Mtok input、cache read $0.3/Mtok 算,一次跑能省约 60% input 成本

### 5.2 目的

给所有固定 system prompt(schema / POC 角色描述 / 7-rule executer prompt)打上
`cache_control: { type: 'ephemeral' }`,降低 token 成本。

**关键风险:Bedrock 对 prompt cache 的支持和直连 Anthropic SDK 不完全对齐**——
必须在本分支内**实测验证**,不支持则降级。

### 5.3 分支信息

- Base:`main`
- 分支名:`perf/prompt-cache`
- PR 大小:中等(主要改 client + 三处 prompt 拼装)
- 完全独立,可与任何其他分支并行

### 5.4 允许碰的文件

- [src/llm/claude-client.ts](../../../src/llm/claude-client.ts) ← 支持 cache_control
- [src/llm/types.ts](../../../src/llm/types.ts) ← system prompt 参数类型扩展
- [src/agents/planner.ts](../../../src/agents/planner.ts) ← schema 段打 cache
- [src/agents/executer.ts](../../../src/agents/executer.ts) ← 7-rule 打 cache
- [src/pipeline/planner.ts](../../../src/pipeline/planner.ts) / [writer.ts](../../../src/pipeline/writer.ts) /
  [storyboarder.ts](../../../src/pipeline/storyboarder.ts) ← v0.2 三阶段 schema 段打 cache
- 新增测试 + 手测脚本

### 5.5 绝对不要碰

- `src/executers/*/tools.ts` 的 POC 实现 ← tier2 分支
- 任何 workspace IO 文件 ← workspace-unification
- LLM tool_use 切换 ← llm-tool-use(本分支只碰 system prompt 打 cache,
  若 llm-tool-use 先 merge,本分支在其基础上 rebase)

### 5.6 具体做法

1. **扩 `chat()` 和 `chatWithTools()` 的 system 参数**
   当前是 `string` 或 `{ role: 'system'; content: string }[]`,扩成支持结构化:

   ```typescript
   type SystemBlock = string | {
     type: 'text';
     text: string;
     cacheControl?: { type: 'ephemeral' };
   };
   type SystemParam = string | SystemBlock | SystemBlock[];
   ```

   SDK 调用时把带 `cacheControl` 的 block 翻译成 `{ type: 'text', text, cache_control: { type: 'ephemeral' } }`。
2. **Planner system prompt 分段**
   `buildPlannerSystemPrompt` 当前一整段,拆成:

   - 段 A(可 cache):TS schema + POC registry + common tools 描述 —— 每次都一样
   - 段 B(不 cache):当前 workspace index + planner memories —— 每轮变

   段 A 打 `cacheControl: { type: 'ephemeral' }`。
3. **Executer system prompt 分段**

   - 段 A(可 cache):7-rule executer prompt + POC 身份描述 + POC tool-set 说明
   - 段 B(不 cache):active_workflow 状态 + task context
4. **v0.2 三阶段 system prompt 分段**

   - 段 A(可 cache):各自的 TS schema 说明
   - 段 B(不 cache):用户具体灵感输入
5. **Bedrock 兼容性验证**

   - 在本分支内写一个最小 smoke:`scripts/prompt-cache-probe.mjs`,用 ≥ 1024 token 的
     system + cache_control,分别在 Bedrock 和直连两路各跑一次,检查响应的 usage 字段
     有没有 `cache_creation_input_tokens` / `cache_read_input_tokens`
   - 若 Bedrock 不支持:
     - `claude-client.ts` 在 `CLAUDE_CODE_USE_BEDROCK=1` 时**自动 strip** `cache_control` 字段,
       不抛错
     - 日志 `[llm] prompt cache disabled on Bedrock transport`
     - 加 env var `CLAUDE_DISABLE_CACHE=1` 做显式总开关
   - 若 Bedrock 支持:两路都开,PR 描述里附实测 usage 对比
6. **usage 透传**
   在 `LlmResponse` / `LlmToolChatResponse` 里透传 `cache_creation_input_tokens` 和
   `cache_read_input_tokens`,logger 打印出来。
7. **< 1024 token 的 prompt 不打 cache**
   Anthropic prompt cache 的最小块是 1024 token,短 prompt 打 cache 是无效且可能报错的。
   `claude-client.ts` 内部做 token 长度估算(粗估 1 token ≈ 4 char),< 1024 自动不加 cache_control。

## 附录 B:分支间协调规则

### B.1 合并地狱预防

| 两分支                                  | 冲突点                                | 解决                                                                      |

### B.2 PR 模板强制字段

每个 PR 描述必须包含:

1. **本分支不碰的文件清单**(从本计划 §x.5 复制)
2. **验收命令**(可直接复制执行)

每月总预算上限 $20(开发期)。每条分支的真 key smoke 单次 ≤ $2。
超支的分支必须在 PR 里解释并提前告知。

### B.4 回滚策略

- 所有分支通过 PR merge,不 force-push main
- 发现合入后有 regression,走新的 `fix/<description>` 分支修,不 revert(避免打断依赖链)
- 只有 Blocker 级 regression 才考虑 `git revert <merge-commit>` + 重做
