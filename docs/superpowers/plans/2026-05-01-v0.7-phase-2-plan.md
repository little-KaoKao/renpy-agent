### 5.3 refactor/storyboarder-enum

#### 5.3.1 背景(为什么要做)

[src/pipeline/coder.ts](../../../src/pipeline/coder.ts) 里的 `pickTransform`(约 L513)用**字符串包含**匹配 LLM 输出:

```typescript
const source = `${transforms} ${staging}`.toLowerCase();
if (source.includes('xxx')) return 'xxx';
// ...
```

同样 `mentions(effects, 'sakura')`(约 L319)靠关键词触发粒子。结果:

- LLM 说 "forehead close-up" → 命中 `forehead`
- LLM 说 "floating through the air" → 关键词没命中,回落 `stand`
- LLM 写中文 "樱花飘落" → 无法触发 sakura 效果(关键词是英文)
- 所有项目都长得像 baiying-demo,transform 多样性极低

**目的**:把 `transforms` / `staging` / `effects` 从自由文本改成 enum,coder 不再做字符串 match。顺带扩充 6 个新 transform。

#### 5.3.2 允许碰的文件

- [src/pipeline/types.ts](../../../src/pipeline/types.ts) ← `StoryboarderOutputShot` 字段改 enum
- [src/pipeline/storyboarder.ts](../../../src/pipeline/storyboarder.ts) ← system prompt 明示 enum;tool schema 用 `enum: [...]`
- [src/pipeline/coder.ts](../../../src/pipeline/coder.ts) ← 去掉 `includes` match,扩 transform 渲染
- [src/executers/storyboarder/tools.ts](../../../src/executers/storyboarder/tools.ts) ← `condense_to_shots` schema 同步
- 对应测试

#### 5.3.3 绝对不要碰

- workspace IO 任何文件
- executer 的底层生成函数
- AssetRegistry
- Planner / Executer loop

#### 5.3.4 具体做法

1. **定义 enum**(in `types.ts`):

   ```typescript
   export const SHOT_TRANSFORMS = [
     'stand', 'lookup', 'front', 'finger', 'forehead', 'heart_pulse', 'reset',
     // 新增:
     'pan_left', 'pan_right', 'fade_in', 'fade_out', 'shake', 'blink',
   ] as const;
   export type ShotTransform = typeof SHOT_TRANSFORMS[number];

   export const SHOT_STAGING = [
     'solo_center', 'solo_left', 'solo_right', 'two_shot', 'group', 'none',
   ] as const;
   export type ShotStaging = typeof SHOT_STAGING[number];

   export const SHOT_EFFECTS = [
     'sakura', 'snow', 'rain', 'lensflare', 'none',
   ] as const;
   export type ShotEffect = typeof SHOT_EFFECTS[number];
   ```

   `StoryboarderOutputShot`:`transforms: string` → `transform: ShotTransform`、`staging: string` → `staging: ShotStaging`、`effects: string` → `effects: ShotEffect[]`(改成数组)。

2. **兼容老 workspace**(**关键,不能删老字段**):
   M5 已产生一份 `runtime/games/smoke-v5/workspace/storyboard.json`,字段全是自由 string。加一个 `normalizeShotFromLegacy(shot)` helper:读入时 string → 尝试匹配 enum,匹配不到 fallback `stand` / `solo_center` / `[]` 并打一次 warn(log dedup,同一字段只 warn 一次)。
   推荐同时把旧字段 rename 成 `*_raw` 保留,新字段(enum 值)并排写,给 rebuild 留一条后路。

3. **Storyboarder system prompt** 列 enum 白名单 + 每项短描述;tool `condense_to_shots.inputSchema` 用 JSON Schema 的 `enum` 字段约束 LLM 输出。

4. **Coder 改精确匹配**:
   - `pickTransform` 直接 `shot.transform`,删除所有 `includes` 匹配
   - effects 改 `shot.effects.includes('sakura')` 数组成员判断
   - `renderTransforms` 扩充 `pan_left` / `pan_right` / `fade_in` / `fade_out` / `shake` / `blink` 的 Ren'Py ATL 定义(参考 `docs/examples/baiying-demo/game/script.rpy` 既有写法)

#### 5.3.5 验收条件

- [ ] `pnpm test` 全绿(M5 基线 386,预计 +5 新测试)
- [ ] Storyboarder 输出的 shot 字段全部合法 enum;schema 非法 enum 直接 LLM retry
- [ ] Coder 代码里 0 处 `includes(...)` 匹配 transform / staging / effects
- [ ] ≥ 3 个新 transform 的渲染分支有单元测试覆盖(`pan_left` / `fade_in` / `shake`)
- [ ] 老 workspace(M5 smoke-v5 产物)能被 `rebuild` 读取,打 warn 但不崩(单元测试模拟即可,不强制真跑)
- [ ] `pnpm typecheck` + `pnpm build` 干净
- [ ] 可选:本地手工 `pnpm start` 跑一份 scripted demo,眼看 .rpy 里 `transform` 定义真的多样化

### 5.5 feat/v5-modify-chain

#### 5.5.1 背景(为什么要做)

V5 架构的**真正卖点**是"改短发"场景:用户说"把女主改成短发",Planner 读 schema 的 `dependencies`,自动算出只要 handoff 角色设计师重生立绘 + `swap_asset_placeholder`,storyboard / script / .rpy 全部不动。

v0.4 的 [modify.ts](../../../src/pipeline/modify.ts) 是**硬编码**的三个函数(改外观 / 改对白 / 重排镜头),扩到"改剧情走向" / "改路线 ending" / "加新角色"都会爆炸。Planner 驱动的修改链条是 V5 对 v0.2 的决定性优势,是 v0.7 的主要交付。

#### 5.5.2 允许碰的文件

- [src/cli.ts](../../../src/cli.ts) ← 新子命令 `v5-modify`
- [src/agents/planner.ts](../../../src/agents/planner.ts) ← 扩 modify-mode 分支(加 `mode: 'generate' | 'modify'` 参数)
- [src/agents/run-v5.ts](../../../src/agents/run-v5.ts) ← 新入口 `runV5Modify`;`RunV5Params` 可选 `modifyIntent: string`
- 新增 e2e 测试(scripted LLM)
- 新增 `docs/v5-modify-chain-trace.md`(Planner trace 示例)
- [src/pipeline/modify.ts](../../../src/pipeline/modify.ts) ← **只加** `@deprecated` JSDoc,保留实现不删

#### 5.5.3 绝对不要碰

- 任何 `src/executers/*` 的实现(tier2 和 task-agent-real 已准备好)
- `src/assets/registry.ts` / `swap.ts`(registry dirty-state 逻辑走 character_designer 的 tool)
- `src/agents/executer.ts` / `common-tools.ts` / `poc-registry.ts` 的既有字段(可加新字段,不改老的)
- v0.4 `modify.ts` 的**实现**(只加 deprecated 注释)

#### 5.5.4 具体做法

1. **CLI 子命令**:`renpy-agent v5-modify <story> "<intent>"`
   - 读 `runtime/games/<story>/`,确认 workspace 存在
   - 调 `runV5Modify({ storyName, modifyIntent, budgetCapUsd, ... })`
   - 结束后同 v5 一样调 Coder + QA 重跑(走 `write_game_project` tool 或复用 v0.4 `rebuild.ts`)

2. **Planner system prompt 扩 modify-mode 分支**
   在 [src/agents/planner.ts](../../../src/agents/planner.ts) 的 system prompt 里加一个可选段(`mode: 'generate' | 'modify'`)。modify 段内容:
   - "你现在处于**修改模式**"
   - 先 `read_from_uri` 探查当前 workspace(已列入 workspace index)
   - 理解用户意图 `<modifyIntent>`
   - 用文档间的 **URI 引用关系**算最小改动集:只 handoff 需要修改的 POC
   - 修改完用 `output_with_finish` 总结改动
   - 加 **3-5 条 few-shot 示例**(改外观 / 改对白 / 加角色 / 换场景)以约束 Planner 的过保守/过激进边界
   - 段 A(7 条基本规则 + few-shot)可缓存,段 B(workspace index + modify intent)随请求变

3. **POC 侧的 dirty-state 支持**
   character_designer 的 `create_or_update_character` tool 在检测到 `visualDescription` 字段变化时,**自动**把 `asset-registry.json` 里该 character 的 `main_image` / `expression` / `dynamic_sprite` 条目的 status 回落到 `placeholder`(记录历史 realAssetLocalPath 但不清零)。下一轮 coder rebuild 时自然走占位,Stage B 异步重生。

4. **`runV5Modify` 顶层 orchestrator**
   [src/agents/run-v5.ts](../../../src/agents/run-v5.ts) 新增:
   - 签名:`runV5Modify({ storyName, modifyIntent, ... })`
   - 复用 `runV5` 的 Planner-Executer 循环,但:
     - `mode: 'modify'` 传给 planner
     - Planner 初始 user prompt 是修改意图
     - 初始 workspace index 是已有 per-URI 文档列表
   - 结束后走 Coder + QA(和 v5 一样)

5. **端到端 scripted 测试**(4 条,**核心验收**):
   - **测试 1 改外观**:intent="把 <某角色> 改成短发" → handoff `character_designer` → 更新 character.json + 清理下游 registry → coder rebuild。**断言:除 character.json 和 asset-registry.json 外,storyboard / script / 其他 character / scene / project / chapter byte-for-byte 不变**
   - **测试 2 改对白**:intent="把 shot 3 line 0 改成 X" → handoff `writer`(或 `storyboarder`,取决于 script vs storyboard 谁 own 这句)→ 只那份文档变
   - **测试 3 加角色**:intent="加一个咖啡店店长" → Planner 自动决定 `producer`(更新 chapter cast)+ `character_designer`(新建 character)+ `writer`(补一句台词)。**断言:原有 character .json 不变**
   - **测试 4 越界保护**:intent="把 shot 99 line 0 改成 X" → Planner 返回 "shot 99 not found" 并 gracefully finish,不崩

6. **Trace 文档**:新建 `docs/v5-modify-chain-trace.md`,贴 4 个测试里 Planner 的 handoff 序列 + 每个 executer 的 taskSummary,参照 M5 report §2 的写法。

7. **标记 v0.4 `modify.ts` deprecated**:三个 export function 加 JSDoc:
   ```
   @deprecated Use `renpy-agent v5-modify` instead. Will be removed in v0.8.
   ```
   不删实现,v0.4 CLI 子命令继续工作。

- [ ] `pnpm test` 全绿,新增 ≥ 4 条 e2e 测试(对应 5.5.4 的测试 1-4)
