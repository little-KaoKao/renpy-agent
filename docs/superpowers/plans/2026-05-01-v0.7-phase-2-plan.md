## 4. 是否用 worktree?

**教训**:Phase 2-A 时 budget-cap 和 M6 smoke 同时在主目录跑,`pnpm build` 产物(dist/)互相污染,M6 smoke 两次失败。**新规则:只要有任意两条工作可能并发,全部上 worktree。**

| 分支 | 用 worktree? | 为什么 |
|---|---|---|
| **prompt-cache-inflation** | **是** | 改 Planner system prompt,可能和 modify-chain/multi-emit 并行 |
| **storyboarder-multi-emit** | **是** | 同上 |
| **qa-read-quota** | **是** | 同上 |
| **task-agent-real** | **是** | 3-5 天工作,新建 3 个文件 + 改测试,开发中会跑很多次 smoke;隔离能避免污染 main 的 dist/ 和 runtime/ |
| **modify-chain** | **是** | 3-5 天 + 依赖 task-agent-real 产物;独立 worktree 方便在其中 `pnpm build && pnpm start` 反复试 |

**Worktree 创建命令**(子会话开头跑):

```powershell
# PowerShell (Windows)
$branch = "refactor/storyboarder-enum"   # 换成你的分支名
$slug = $branch -replace '^(refactor|feat|fix|chore)/', ''
cd E:\RenPy
git fetch origin
git worktree add ".worktrees\$slug" -b $branch origin/main
cd ".worktrees\$slug"
pnpm install   # node_modules 不共享,必须重装
```

```bash
# bash (macOS/Linux 参考)
branch="refactor/storyboarder-enum"
slug=${branch#*/}
cd ~/RenPy
git fetch origin
git worktree add ".worktrees/$slug" -b "$branch" origin/main
cd ".worktrees/$slug"
pnpm install
```

**Worktree 清理**(合并后):
```powershell
cd E:\RenPy
git worktree remove ".worktrees\<slug>"
git branch -d <branch>           # 本地
git push origin --delete <branch> # 远程
```

### 5.3 refactor/storyboarder-enum

#### 5.3.1 背景(为什么要做)

[src/pipeline/coder.ts](../../../src/pipeline/coder.ts) 里的 `pickTransform`(约 L513)用**字符串包含**匹配 LLM 输出:

```typescript
const source = `${transforms} ${staging}`.toLowerCase();
if (source.includes('xxx')) return 'xxx';
// ...
```

同样 `mentions(effects, 'sakura')`(约 L319)靠关键词触发粒子。结果:

- LLM 说 "forehead close-up" → 命中 `forehead`
- LLM 说 "floating through the air" → 关键词没命中,回落 `stand`
- LLM 写中文 "樱花飘落" → 无法触发 sakura 效果(关键词是英文)
- 所有项目都长得像 baiying-demo,transform 多样性极低

**目的**:把 `transforms` / `staging` / `effects` 从自由文本改成 enum,coder 不再做字符串 match。顺带扩充 6 个新 transform。

#### 5.3.2 允许碰的文件

- [src/pipeline/types.ts](../../../src/pipeline/types.ts) ← `StoryboarderOutputShot` 字段改 enum
- [src/pipeline/storyboarder.ts](../../../src/pipeline/storyboarder.ts) ← system prompt 明示 enum;tool schema 用 `enum: [...]`
- [src/pipeline/coder.ts](../../../src/pipeline/coder.ts) ← 去掉 `includes` match,扩 transform 渲染
- [src/executers/storyboarder/tools.ts](../../../src/executers/storyboarder/tools.ts) ← `condense_to_shots` schema 同步
- 对应测试

#### 5.3.3 绝对不要碰

- workspace IO 任何文件
- executer 的底层生成函数
- AssetRegistry
- Planner / Executer loop

#### 5.3.4 具体做法

1. **定义 enum**(in `types.ts`):

   ```typescript
   export const SHOT_TRANSFORMS = [
     'stand', 'lookup', 'front', 'finger', 'forehead', 'heart_pulse', 'reset',
     // 新增:
     'pan_left', 'pan_right', 'fade_in', 'fade_out', 'shake', 'blink',
   ] as const;
   export type ShotTransform = typeof SHOT_TRANSFORMS[number];

   export const SHOT_STAGING = [
     'solo_center', 'solo_left', 'solo_right', 'two_shot', 'group', 'none',
   ] as const;
   export type ShotStaging = typeof SHOT_STAGING[number];

   export const SHOT_EFFECTS = [
     'sakura', 'snow', 'rain', 'lensflare', 'none',
   ] as const;
   export type ShotEffect = typeof SHOT_EFFECTS[number];
   ```

   `StoryboarderOutputShot`:`transforms: string` → `transform: ShotTransform`、`staging: string` → `staging: ShotStaging`、`effects: string` → `effects: ShotEffect[]`(改成数组)。

2. **兼容老 workspace**(**关键,不能删老字段**):
   M5 已产生一份 `runtime/games/smoke-v5/workspace/storyboard.json`,字段全是自由 string。加一个 `normalizeShotFromLegacy(shot)` helper:读入时 string → 尝试匹配 enum,匹配不到 fallback `stand` / `solo_center` / `[]` 并打一次 warn(log dedup,同一字段只 warn 一次)。
   推荐同时把旧字段 rename 成 `*_raw` 保留,新字段(enum 值)并排写,给 rebuild 留一条后路。

3. **Storyboarder system prompt** 列 enum 白名单 + 每项短描述;tool `condense_to_shots.inputSchema` 用 JSON Schema 的 `enum` 字段约束 LLM 输出。

4. **Coder 改精确匹配**:
   - `pickTransform` 直接 `shot.transform`,删除所有 `includes` 匹配
   - effects 改 `shot.effects.includes('sakura')` 数组成员判断
   - `renderTransforms` 扩充 `pan_left` / `pan_right` / `fade_in` / `fade_out` / `shake` / `blink` 的 Ren'Py ATL 定义(参考 `docs/examples/baiying-demo/game/script.rpy` 既有写法)

#### 5.3.5 验收条件

- [ ] `pnpm test` 全绿(M5 基线 386,预计 +5 新测试)
- [ ] Storyboarder 输出的 shot 字段全部合法 enum;schema 非法 enum 直接 LLM retry
- [ ] Coder 代码里 0 处 `includes(...)` 匹配 transform / staging / effects
- [ ] ≥ 3 个新 transform 的渲染分支有单元测试覆盖(`pan_left` / `fade_in` / `shake`)
- [ ] 老 workspace(M5 smoke-v5 产物)能被 `rebuild` 读取,打 warn 但不崩(单元测试模拟即可,不强制真跑)
- [ ] `pnpm typecheck` + `pnpm build` 干净
- [ ] 可选:本地手工 `pnpm start` 跑一份 scripted demo,眼看 .rpy 里 `transform` 定义真的多样化

#### 5.3.6 风险

| 风险 | 兜底 |
|---|---|
| LLM 输出非 enum 值 | tool schema `enum` 约束(Bedrock SDK 支持);万一失败,Executer 层 retry 一次 |

### 5.5 feat/v5-modify-chain

#### 5.5.1 背景(为什么要做)

V5 架构的**真正卖点**是"改短发"场景:用户说"把女主改成短发",Planner 读 schema 的 `dependencies`,自动算出只要 handoff 角色设计师重生立绘 + `swap_asset_placeholder`,storyboard / script / .rpy 全部不动。

v0.4 的 [modify.ts](../../../src/pipeline/modify.ts) 是**硬编码**的三个函数(改外观 / 改对白 / 重排镜头),扩到"改剧情走向" / "改路线 ending" / "加新角色"都会爆炸。Planner 驱动的修改链条是 V5 对 v0.2 的决定性优势,是 v0.7 的主要交付。

#### 5.5.2 允许碰的文件

- [src/cli.ts](../../../src/cli.ts) ← 新子命令 `v5-modify`
- [src/agents/planner.ts](../../../src/agents/planner.ts) ← 扩 modify-mode 分支(加 `mode: 'generate' | 'modify'` 参数)
- [src/agents/run-v5.ts](../../../src/agents/run-v5.ts) ← 新入口 `runV5Modify`;`RunV5Params` 可选 `modifyIntent: string`
- 新增 e2e 测试(scripted LLM)
- 新增 `docs/v5-modify-chain-trace.md`(Planner trace 示例)
- [src/pipeline/modify.ts](../../../src/pipeline/modify.ts) ← **只加** `@deprecated` JSDoc,保留实现不删

#### 5.5.3 绝对不要碰

- 任何 `src/executers/*` 的实现(tier2 和 task-agent-real 已准备好)
- `src/assets/registry.ts` / `swap.ts`(registry dirty-state 逻辑走 character_designer 的 tool)
- `src/agents/executer.ts` / `common-tools.ts` / `poc-registry.ts` 的既有字段(可加新字段,不改老的)
- v0.4 `modify.ts` 的**实现**(只加 deprecated 注释)

#### 5.5.4 具体做法

1. **CLI 子命令**:`renpy-agent v5-modify <story> "<intent>"`
   - 读 `runtime/games/<story>/`,确认 workspace 存在
   - 调 `runV5Modify({ storyName, modifyIntent, budgetCapUsd, ... })`
   - 结束后同 v5 一样调 Coder + QA 重跑(走 `write_game_project` tool 或复用 v0.4 `rebuild.ts`)

2. **Planner system prompt 扩 modify-mode 分支**
   在 [src/agents/planner.ts](../../../src/agents/planner.ts) 的 system prompt 里加一个可选段(`mode: 'generate' | 'modify'`)。modify 段内容:
   - "你现在处于**修改模式**"
   - 先 `read_from_uri` 探查当前 workspace(已列入 workspace index)
   - 理解用户意图 `<modifyIntent>`
   - 用文档间的 **URI 引用关系**算最小改动集:只 handoff 需要修改的 POC
   - 修改完用 `output_with_finish` 总结改动
   - 加 **3-5 条 few-shot 示例**(改外观 / 改对白 / 加角色 / 换场景)以约束 Planner 的过保守/过激进边界
   - 段 A(7 条基本规则 + few-shot)可缓存,段 B(workspace index + modify intent)随请求变

3. **POC 侧的 dirty-state 支持**
   character_designer 的 `create_or_update_character` tool 在检测到 `visualDescription` 字段变化时,**自动**把 `asset-registry.json` 里该 character 的 `main_image` / `expression` / `dynamic_sprite` 条目的 status 回落到 `placeholder`(记录历史 realAssetLocalPath 但不清零)。下一轮 coder rebuild 时自然走占位,Stage B 异步重生。

4. **`runV5Modify` 顶层 orchestrator**
   [src/agents/run-v5.ts](../../../src/agents/run-v5.ts) 新增:
   - 签名:`runV5Modify({ storyName, modifyIntent, ... })`
   - 复用 `runV5` 的 Planner-Executer 循环,但:
     - `mode: 'modify'` 传给 planner
     - Planner 初始 user prompt 是修改意图
     - 初始 workspace index 是已有 per-URI 文档列表
   - 结束后走 Coder + QA(和 v5 一样)

5. **端到端 scripted 测试**(4 条,**核心验收**):
   - **测试 1 改外观**:intent="把 <某角色> 改成短发" → handoff `character_designer` → 更新 character.json + 清理下游 registry → coder rebuild。**断言:除 character.json 和 asset-registry.json 外,storyboard / script / 其他 character / scene / project / chapter byte-for-byte 不变**
   - **测试 2 改对白**:intent="把 shot 3 line 0 改成 X" → handoff `writer`(或 `storyboarder`,取决于 script vs storyboard 谁 own 这句)→ 只那份文档变
   - **测试 3 加角色**:intent="加一个咖啡店店长" → Planner 自动决定 `producer`(更新 chapter cast)+ `character_designer`(新建 character)+ `writer`(补一句台词)。**断言:原有 character .json 不变**
   - **测试 4 越界保护**:intent="把 shot 99 line 0 改成 X" → Planner 返回 "shot 99 not found" 并 gracefully finish,不崩

6. **Trace 文档**:新建 `docs/v5-modify-chain-trace.md`,贴 4 个测试里 Planner 的 handoff 序列 + 每个 executer 的 taskSummary,参照 M5 report §2 的写法。

7. **标记 v0.4 `modify.ts` deprecated**:三个 export function 加 JSDoc:
   ```
   @deprecated Use `renpy-agent v5-modify` instead. Will be removed in v0.8.
   ```
   不删实现,v0.4 CLI 子命令继续工作。

## 8. feat/prompt-cache-inflation(M7 Major-1 修复)

### 8.1 背景

M7 smoke(2026-05-02)证实:`fix/bedrock-cache-default` 已经合入 main,`cache_control: {type:'ephemeral'}` 确实发给了 Bedrock,但 **3 次连续 smoke(M5 / M6-pre-fix / M7)的 `cacheCreation` 和 `cacheRead` 都是 0**。

根因(实测确认):
- `PLANNER_SYSTEM_PROMPT`: **1408 chars**(约 352 tokens)
- Executer cacheable segment: **977 chars**(约 244 tokens)
- Anthropic prompt cache 最小单元:**1024 tokens(~4096 chars)**
- SDK 在 token 不够时**静默忽略** `cache_control`

结论:"开启 cache" 的 fix 生效了,但静态段不够长,cache 永远不激活。

### 8.2 目的

把 Planner / Executer 的 `cacheControl: ephemeral` 标记的那段 system prompt **膨胀到 > 4096 chars**,让 Anthropic 真的缓存,V5 smoke 单次再省 30-50% input tokens。

### 8.3 允许碰的文件

- `src/agents/planner.ts`(膨胀 `PLANNER_SYSTEM_PROMPT`)
- `src/agents/executer.ts`(膨胀 cacheable segment,不碰 tool schema)
- `src/schema/galgame-workspace.ts`(**只 import/re-export 常量**,不改字段定义)
- `src/agents/poc-registry.ts`(可选:更详细的 POC description 文案)
- 对应测试

### 8.4 绝对不要碰

- `src/llm/claude-client.ts` 里 `PROMPT_CACHE_MIN_CHARS` / `shouldApplyCacheControl` / `bedrockCacheExplicitOff` 任何逻辑 —— 这些门槛设计是对的,别碰
- 任何 POC 的 tools.ts
- 任何 common-tools.ts 的行为
- workspace IO

### 8.5 具体做法

1. **在 `galgame-workspace.ts` 末尾添加 `SCHEMA_DIGEST` 常量**
   内容:一段 ~3000 char 的人类可读描述,把 19 个 DocumentKind + 关键字段 + 典型 URI 形式列出来。参照 PLAN.md 已删除的 §6 写法:
   ```
   DocumentKind: project | chapter | character | scene | ...(19 种)

   URI 形式: workspace://<kind>[/<slug>]
     singleton: workspace://project, workspace://chapter, workspace://script, workspace://storyboard
     collection: workspace://character/<slug>, workspace://scene/<slug>, ...

   字段间引用关系:
     - Storyboard.shots[i] 引用 characterUri / sceneUri(不复制数据)
     - Script.scenes[i].characters 是 character name 数组(不是 URI)
     - RpyFile 引用 storyboardUri + assetRegistryUri
     ...
   ```

   **不要** import TypeScript 源码字面量,因为那会把开发注释也塞进去。**手写一份精简的、Planner 需要知道的字段说明**,放常量里。

2. **planner.ts 的 PLANNER_SYSTEM_PROMPT 膨胀**
   原来的 7 条 rule 保留。在 rule 8(短路 hint)前面加一段:
   ```
   ---
   Workspace schema reference (for your planning decisions):

   ${SCHEMA_DIGEST}

   ---
   POC capabilities in detail:

   - producer: owns project / chapter. Tools: create_project, create_chapter.
   - character_designer: owns character documents + main image / expression / dynamic sprite registry entries. Tools: create_or_update_character, generate_character_main_image (via task_agent), ...
   - scene_designer: ...
   - writer: ...
   - storyboarder: ...
   - coder: ...
   - qa: ...
   ---
   ```

   目标:让 `PLANNER_SYSTEM_PROMPT.length ≥ 5000 chars`(给 CJK-heavy 预留余量过 4096 门槛)。

3. **executer.ts 的 cacheable segment 同样膨胀**
   把 V5 的 7 条 Executer 规则(见 PLAN design spec §2.1)明写出来,每条 150-300 char,加 1-2 个例子。目标:`cacheableSystemPrompt.length ≥ 5000 chars`。

4. **不要** 把 POC 的 tool schemas 塞进去 —— 那些是动态的,每个 POC 不一样,塞进来反而打破 cache。

### 8.6 验收条件

- [ ] `pnpm test` 全绿(基线 416,预计 +3-5 新测:asserts PLANNER_SYSTEM_PROMPT.length ≥ 5000,同样的 for executer)
- [ ] **本地可验证(不花钱)**:写一个 debug 脚本打印两者长度
- [ ] 真 key 验收(预算 $2-3):
  - 清 workspace,`pnpm build`,跑 `scripts/v5-real-key-smoke.mjs --budget-cap 3`
  - 期望:`cacheCreationInputTokens > 0` 在至少第 2 次 LLM call 出现
  - 期望:`cacheReadInputTokens > 0` 在第 3 次或之后出现(5 min TTL 内同一段)
  - 期望:`cacheReadShareOfInput ≥ 25%`(Planner + Executer 都有 cache hit)
  - 期望:Run A 成本 ≤ $2.5(M7 是 $2.89,预期降 15-30%)
- [ ] `docs/smoke-reports/v5-real-key-smoke-<date>-cache-inflation.md` 落地,对比 M7

### 8.7 风险

| 风险 | 兜底 |
|---|---|
| 膨胀后 LLM 被"淹死"(context 爆或者 POC description 冲淡 rules) | 膨胀内容放在 rule 之**后**,rules 保持前置;加测试断言 rules 7 条完整未被挤出 |
| CJK 字符 token 密度高,4096 char 仍不够 | 过 5000 char 给余量;如果实测仍 miss,加到 6000 |

## 9. fix/storyboarder-multi-emit(M7 Major-2 修复)

### 9.1 背景

M7 smoke trace 显示 `condense_to_shots` 被调 **3 次**(对应 `emit_storyboarder_output` 3 次),占了 21% 的总 input token(132k)。第二次调用 `cgEntries=0`(退步),第三次才恢复 5。

这不是 `runStoryboarder` 内部的 retry(已经由 `7ecab13` 约束成 2 次最多)。**是 Executer 层的 LLM 主动决定"再调一次同样的 tool"。** 表面上 LLM 认为多调可以"精修",实际上每次都覆盖 workspace://storyboard,浪费且产生中间态。

### 9.2 目的

给 Executer 循环加一个 per-tool-name 的软上限:同一 POC 在同一次 handoff 内,同名 tool 被调 ≥ 3 次时 **拒绝执行并返 error**,让 LLM 停手。

### 9.3 允许碰的文件

- `src/agents/executer.ts`(加 tool-call 计数)
- 对应测试

### 9.4 绝对不要碰

- `src/agents/planner.ts` / `common-tools.ts` / `poc-registry.ts`
- 任何 POC 的 tools.ts

### 9.5 具体做法

1. **在 `runExecuterTask` 循环内部维护 `toolCallCounts: Map<string, number>`**
2. **每次 dispatch tool 前检查**:
   ```typescript
   const n = (toolCallCounts.get(tu.name) ?? 0) + 1;
   toolCallCounts.set(tu.name, n);
   if (n > 2) {
     return {
       toolUseId: tu.id,
       content: JSON.stringify({
         error: `tool "${tu.name}" called ${n} times in this handoff; refusing to execute. Call output_with_finish now.`,
         retry: false,
         guidance: `You have already invoked ${tu.name} ${n-1} times. Further invocations will not be executed. Call output_with_finish with a summary of what was accomplished, even if partial.`,
       }),
       isError: true,
     };
   }
   ```
3. **豁免**:`read_from_uri`、`output_with_plan`、`output_with_finish` 不算(这三个语义上合法重复)。白名单常量在同文件。
4. **Logger**:触发时 `ctx.logger.warn('executer.soft_limit_hit', { tool: tu.name, count: n })`

### 9.6 验收条件

- [ ] `pnpm test` 全绿(+2-3 测试:同名 tool 第 3 次拒绝、白名单 tool 不受限、reject 后 LLM 调 finish 能正常结束)
- [ ] 测试用 scripted LLM 模拟"第一轮调 condense_to_shots,第二轮再调,第三轮再调"场景,断言第三轮收到 error + 不实际执行 runStoryboarder
- [ ] 不需要真 key 验证(M7 trace 足以证明有必要)

### 9.7 风险

| 风险 | 兜底 |
|---|---|
| 软上限打断合法场景(LLM 真的需要多次 retry 同一个) | 内部重试已经被 `7ecab13` 约束;外部软上限 = 2 对正常流程足够;如果未来某 POC 真需要,加到白名单 |
| 计数器跨 handoff 泄漏 | 用函数局部变量,每次 `runExecuterTask` 新建 |

### 9.8 预计工作量

1 天(本地开发 + 测试)。

## 10. feat/qa-read-quota(M7 Minor-3 修复)

### 10.1 背景

M7 smoke 里 QA 只调了 6 次 `read_from_uri`,1 次 `run_qa`,立刻 `output_with_finish`。M5 当时 QA 读了 43 次做 cross-ref,抓出 3 BLOCKING + 12 其他 issue。M7 QA "轻松 PASS" 更可能是**深度不够的假 PASS**,不是真的没问题。

lint 只校验 Ren'Py 语法;跨文档引用(角色在 storyboard 出现但 character 没这个 name 之类)lint 看不到。

### 10.2 目的

QA Executer 强制在 `run_qa` 之前读到 workspace 文档的一定比例,否则拒绝 finish。

### 10.3 允许碰的文件

- `src/executers/qa/tools.ts`(加 read 计数 + min-read 校验)
- `src/agents/poc-registry.ts`(qa 的 description 加说明)
- 对应测试

### 10.4 绝对不要碰

- 其他 POC 的 tools.ts
- Planner / common-tools
- run_qa 本身的 Ren'Py lint 调用逻辑

### 10.5 具体做法

1. **`run_qa` executor 开头做 workspace 统计**:
   ```typescript
   const index = await buildWorkspaceIndex(ctx.gameDir);
   const totalDocs = index.entries.length;
   const minRequiredReads = Math.max(5, Math.ceil(totalDocs * 0.5)); // 至少一半
   ```
2. **在 Executer 层给 QA ctx 注入 `readFromUriCount`**(需要和 §9 相同机制,或独立一个小计数器)
3. **`run_qa` 前校验**:count < minRequired 时返 `{ error, retry: false, guidance: "read at least N docs via read_from_uri before running QA lint" }`
4. **poc-registry qa description 加一句**:
   > QA 在跑 lint 前必须 read 所有关键文档(character × scene × storyboard × script)做 cross-check,否则 run_qa 会拒绝执行。

### 10.6 验收条件

- [ ] `pnpm test` 全绿(+2 测试:workspace 10 文档时 QA 读 4 次后 run_qa 被拒;QA 读 8 次后允许 run_qa)
- [ ] 独立的 scripted e2e:让 scripted QA 不读任何 URI 直接调 run_qa,期望收到 error
- [ ] 不需要真 key

### 10.7 风险

| 风险 | 兜底 |
|---|---|
| 计数机制和 §9 的冲突 | 优先合 §9 的 executer 改动,本分支 rebase;或者两者 rebase 到 main 后合并计数机制 |
| 小项目 QA 读 4 个文档够了,强制读 5 过度 | `Math.max(5, totalDocs * 0.5)` —— 本来就有地板,不会在 8 文档项目强制读 8 |

### 10.8 预计工作量

1 天(本地开发 + 测试)。
