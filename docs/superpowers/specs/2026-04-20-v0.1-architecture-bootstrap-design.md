## 1. TS 工具链(§1)

### 1.1 为什么要

- `schema/` 和 `workflows/` 是 **prompt 输入**(LLM 读 TS 比读自然语言稳),**不会被运行**;但我们需要 `tsc --noEmit` 保证它们语法对、类型自洽 —— 否则喂给 Planner 的 prompt 里有坏类型,调试成本极高
- [executers/common/runninghub-client.ts](../../../executers/common/runninghub-client.ts) 和 [scripts/smoke-runninghub.ts](../../../scripts/smoke-runninghub.ts) **是要跑的**,需要一个能直接执行 TS 的运行器

### 1.2 技术栈

| 项          | 选择                                   | 原因                                                                                   |
| ----------- | -------------------------------------- | -------------------------------------------------------------------------------------- |
| 包管理器    | **pnpm**(经 fnm 的 Node 装出来)      | 用户本机已有(记忆 `user_local_toolchain`),磁盘 / 安装速度都比 npm/yarn 好            |
| Node 版本   | **20 LTS**                             | 最稳,[.node-version](../../../.node-version) 钉住                                                      |
| TS 版本     | **TypeScript 5.x**(最新稳定)        | ESM / `moduleResolution: "bundler"` 支持完整                                           |
| 运行器      | **tsx**                                | 直接 `tsx script.ts` 就跑,Windows ESM 路径不折腾                                      |
| 类型检查    | **tsc --noEmit**                       | 只做类型检查,不产 `dist/`(本期没有构建产物)                                         |
| 环境变量    | **dotenv**                             | smoke 脚本读 `.env` 里的 `RUNNINGHUB_API_KEY`                                          |
| Module 体系 | **ESM**(`"type": "module"`)          | 新项目默认 ESM,未来更少迁移成本                                                       |
| 模块解析    | **`moduleResolution: "bundler"`**      | 不强制 `.js` 后缀,Windows 上开发体验最好;我们本来就不构建到 `.js`                    |

## 2. Workspace schema(§2)

### 2.1 文件

[schema/galgame-workspace.ts](../../../schema/galgame-workspace.ts)

### 2.2 目的

- 作为 **Planner 的 system prompt 输入**:LLM 读这份 TS 理解 workspace 有哪些文档、字段、以及跨文档依赖
- 作为**编辑期契约**:执行层(executers)写入 workspace 时按这些类型构造

### 2.3 顶层结构(散文 + 区块注释)

文件最开头用多行注释写一段**散文说明**,直接面向读它的 LLM:
- workspace 是什么(15 个按 URI 引用的文档)
- URI 作用(跨文档引用不拷数据,资产换了上游无感)
- Stage A / Stage B 切换的钩子在 AssetRegistry
- `dependencies` JSDoc 是 Planner 算级联的依据

之后按 POC 分区,每个区一个 `// region XX` 注释,内部放该 POC own 的 document interface。

### 2.4 基础类型

```ts
export type Uri = string; // "workspace://role/baiying" / "workspace://scene/chapel-night"
export type Timestamp = string; // ISO 8601

export type DocStatus = "draft" | "ready" | "stale";

export interface DocBase {
  uri: Uri;
  title: string;
  status: DocStatus;
  createdAt: Timestamp;
  updatedAt: Timestamp;
}
```

- **URI scheme**:`workspace://` (PLAN §9 决策)。URI 类型不做 branded type(保持简单,LLM 读 TS 时别被复杂泛型绕晕)
- **`status`**:`stale` = 依赖上游变过、自身还没重生

### 2.5 15 个文档(按 POC 分区)

每个 interface extends `DocBase`,并在 TSDoc 顶部标 `@dependencies` 列出它引用的上游文档类型 + 字段名。Planner 读这段就能算出"改了 X 要级联重生哪些"。

#### 制作人(5)

1. `Inspiration` —— 用户原始输入(文字 / 图片 URL);无依赖
2. `Project` —— 项目定位、题材、画风;deps: `inspirationUri`
3. `Chapter` —— 章节骨架;deps: `projectUri`
4. `Route` —— 路线定义;deps: `projectUri`
5. `Ending` —— 结局;deps: `routeUri`

#### 编剧(1)

6. `Script` —— 章节级对白脚本;deps: `chapterUri`, `characterUris[].voiceTag`

#### 角色设计师(1)

7. `Character` —— 外观描述 + `mainImageUri`(立绘主图)+ `expressionDifferentials[]`(表情差分)+ 可选 `dynamicPortraitUri`(视频循环);deps: 无外部,但动态立绘依赖自身 `mainImageUri`

#### 场景设计师(2)

8. `Scene` —— 背景图 + 时段/光照变体;无依赖
9. `Prop` —— 道具图;deps: `sceneUri`(可选)

#### 分镜师(2)

10. `Storyboard` —— `Shot[]`,镜头级演出方案;deps: `scriptUri`, `characterUris[]`, `sceneUris[]`
11. `Cutscene` —— 视频镜头(过场 CG、关键剧情 CG);deps: `storyboardUri`, `characterUris[]`, `sceneUris[]`

#### Ren'Py 编码师(2)

12. `RpyFile` —— 生成的 `.rpy` 文件索引;deps: `storyboardUri`, `cutsceneUris[]`, `assetRegistryUri`
13. `AssetRegistry` —— `placeholderId → realAssetUri | null` 映射表,Stage A/B 切换的账本;deps: `rpyFileUri`

#### QA(2)

14. `TestRun` —— 一次跑测结果;deps: `rpyFileUri`
15. `BugReport` —— 单个 bug 条目 + 定位;deps: `testRunUri`

### 2.6 placeholder / ticket 合约

生成类字段(`mainImageUri`、`backgroundUri`、`cutsceneVideoUri` 等)都是 `Uri` 类型,且 **URI 本身编码了 Stage**:
- `workspace://placeholder/solid-blue-square/xxx` —— Stage A 占位
- `workspace://asset/runninghub/<task-id>/<file>` —— Stage B 真资产

RunningHub 任务 ID 单独保存在该文档的 `generationTicket?: { taskId: string; submittedAt: Timestamp; webappId: string }` 字段,不入 URI。

### 2.7 导出

文件末尾导出联合类型 `export type WorkspaceDoc = Inspiration | Project | ... | BugReport`,给 `read_from_uri` 的返回做类型窄化。

## 3. POC workflows(§3)

### 3.1 文件

[workflows/galgame-workflows.ts](../../../workflows/galgame-workflows.ts)

### 3.2 目的

- **纯签名文件**:所有函数用 `export declare function`,没有实现体
- 这份 TS 塞进 Planner 的 system prompt,让 LLM 知道可以调哪些 tool、每个 tool 要什么参数、返回什么
- 实现散落在 [executers/](../../../executers/)(v0.2 起逐步填)

### 3.3 结构

文件按 **8 个 common + 7 位 POC** 分区:

```ts
// region Common Tools
export declare function output_with_plan(input: OutputWithPlanInput): Promise<OutputWithPlanOutput>;
// ... 7 more

// region Producer(制作人)
export declare function create_or_update_project(input: ...): Promise<...>;
// ... 4 more

// region Writer(编剧)
// ...
```

### 3.4 函数清单(39 个)

**Common Tools(8)** —— 来自 HOGI V5 命名,PLAN §4.1:

5. `handoff_to_agent(agentId: AgentId)` —— 切身份到某 POC(`AgentId` 是联合字面量)
6. `check_workflow_params(workflowId: string)` —— 校验上下文满足参数
7. `call_task_agent(agentName: string, input: Record<string, unknown>)` —— 调 HOGI 既有 task agent(如"角色生图词扩写助理")
8. `get_workflow_guide(topic: string)` —— 读 workflow 的 guide 文档

**制作人(5)**:

9. `create_or_update_project`
10. `create_or_update_chapter`
11. `create_or_update_route`

**编剧(3)**:

**角色设计师(5)**:

**场景设计师(4)**:

**分镜师(5)**:

**Ren'Py 编码师(5)**:

**QA 测试员(4)**:

36. `run_game(rpyFileUri: Uri, scenario?: string)` —— 跑 `renpy-sdk/renpy.exe`,产 TestRun
37. `capture_bug_report(testRunUri: Uri, bug: BugDraft)`
38. `kick_back_to_coder(bugReportUri: Uri)` —— 把 bug 推回编码师,自己不改代码
39. `list_test_runs(rpyFileUri: Uri)`

### 3.5 QA 只读性的物理保障

PLAN §5 规定 QA 不能直接写 `.rpy`。**本期通过 tool-set 物理隔离来实现**:QA 身份下没有 `compile_rpy_*` / `swap_asset_*`,只能 `kick_back_to_coder`。这比 prompt 层面的 "不要改代码" 可靠得多。

### 3.6 Input/Output 类型命名约定

每个函数都有 `XxxInput` / `XxxOutput` 独立 interface,不用 inline 参数。理由:

- LLM 读到参数名就明白语义
- 未来扩字段不动函数签名
- `Output` 类型里统一用 `{ uri: Uri; placeholderUri?: Uri; ticket?: GenerationTicket }` 三元组,所有生成类函数看起来一致

### 3.7 类型导入

`workflows/galgame-workflows.ts` 从 [schema/galgame-workspace.ts](../../../schema/galgame-workspace.ts) import 所需类型(`Uri`、`Character`、`WorkspaceDoc` 等),不重复定义。

## 4. RunningHub 客户端 + smoke(§4)

### 4.1 文件

- [executers/common/runninghub-client.ts](../../../executers/common/runninghub-client.ts) —— 客户端类
- [scripts/smoke-runninghub.ts](../../../scripts/smoke-runninghub.ts) —— 最小握手脚本

### 4.2 客户端 API

```ts
export interface RunningHubClientOptions {
  apiKey: string;
  baseUrl?: string;
  fetchImpl?: typeof fetch;
}

export interface RunningHubTask {
  taskId: string;
  status: "pending" | "running" | "succeeded" | "failed";
  outputs?: RunningHubOutput[];
  error?: { code: string; message: string };
}

export interface RunningHubOutput {
  type: "image" | "video";
  url: string;
  mime: string;
}

export interface RunningHubSubmitInput {
  webappId: string; // 如 "api-425766740"(悠船文生图-v7)
  nodeInputs: Record<string, unknown>;
}

export class RunningHubClient {
  constructor(opts: RunningHubClientOptions);
  submit(input: RunningHubSubmitInput): Promise<RunningHubTask>;
  getStatus(taskId: string): Promise<RunningHubTask>;
  run(input: RunningHubSubmitInput, pollIntervalMs?: number): Promise<RunningHubTask>;
}
```

- **`submit`**:提交任务,拿 `taskId`。对应 Stage A 占位 + ticket 流程
- **`getStatus`**:单次查询,给 Stage B 轮询/事件回写用
- **`run`**:`submit + 轮询 getStatus 直到终态`。smoke 脚本走这个

### 4.3 `TODO: RH-API-CONFIRM`(**本期的主要风险**)

RunningHub 的 HTTP API 细节(endpoint 路径、请求 body schema、轮询频率上限、鉴权方式)**我们还没核实过**。本期处理策略:

1. 客户端**写出完整类型 + 方法签名**,body 组装逻辑先按合理猜测实现
2. 所有未确认的 endpoint / 字段,**在代码里用 `// TODO: RH-API-CONFIRM` 明确标出**
3. smoke 脚本**默认不真发 POST**:先跑"构造 body + 打印出来 + 打印 apiKey 前 4 位",验证本地环境装配对了
4. 加一个环境变量开关 `RH_LIVE=1`,置位才真调 API
5. v0.2 前读 RunningHub 官方文档(或跑一次真调)把 `// TODO: RH-API-CONFIRM` 清零

这个姿态避免了"没 API 文档就卡住",也避免了"猜错 API 但没标记,后期谁都不知道哪里猜的"。

### 4.4 smoke 脚本

```
scripts/smoke-runninghub.ts
```

内容逻辑:

1. `dotenv.config()` 读 `.env`
2. `process.env.RUNNINGHUB_API_KEY` 缺失就非零退出,打印"去 .env 填"
3. new RunningHubClient,hardcode `webappId: "api-425766740"`(悠船文生图-v7)+ `prompt: "a solid blue placeholder square"`
4. 默认走"dry run"分支:打印组装好的 request body + redacted apiKey
5. `RH_LIVE=1` 时走 `client.run()`,打印最终 task 结果(taskId / status / outputs)

### 4.5 验收

- `pnpm check` 通过
- `pnpm smoke:runninghub`(无 `.env`)→ 清晰的错误信息
- `pnpm smoke:runninghub`(有 `.env`,无 `RH_LIVE`)→ 打印 dry-run body 成功
- `RH_LIVE=1 pnpm smoke:runninghub`:**本期不强制通过**,但跑起来应该只会 API 响应错误,不会本地异常
