## 1. 模块结构

### 1.1 新增顶层目录 `src/agents/`

V5 专属,和 `src/pipeline/`(串行世界)**并列**。V5 语义上不是"pipeline 的一部分",而是"pipeline 的上层"—— Planner 通过 handoff 驱动 POC,POC 内部的生成函数复用 `src/pipeline/*` 和 `src/executers/*/generate-*.ts` 原件。

```
src/agents/
├── planner.ts              # Planner LLM loop:system prompt + output_with_plan/finish/read_from_uri
├── executer.ts             # Executer LLM loop:统一 7-rule prompt + tool-set 动态装载
├── common-tools.ts         # 5 个 common tool 的 schema + 实现
├── common-tools.test.ts
├── poc-registry.ts         # POC → tool-set 映射表(身份靠 tool-set 区分的源头)
├── memory.ts               # planner_memories JSONL 读写 + prepend 助手
├── memory.test.ts
├── workspace-index.ts      # 从 workspace/*.json 生成 {uri, title, status, kind} 扁平索引
├── workspace-index.test.ts
├── run-v5.ts               # v5 orchestrator(等价于 run-pipeline.ts 在 v5 世界的角色)
└── run-v5.test.ts
```

### 1.2 扩展现有 POC 目录

每位 POC 加一个 `tools.ts`,把该 POC 已有的 `generate-*.ts` 函数封装成 Executer 能看到的 tool schema。举例:

```
src/executers/character-designer/
├── generate-main-image.ts     (已存在,不动)
├── generate-expression.ts     (已存在,不动)
├── tools.ts                   (新增:character_designer 的 tool-set)
└── tools.test.ts
```

11 位 POC 全部建 `tools.ts`(见 §3);制作人 POC 当前没目录,新建 `src/executers/producer/`。

### 1.3 LLM client 扩展

`src/llm/claude-client.ts` 的 `ClaudeLlmClient.chat()` 目前只接 text。V5 需要 `tool_use` / `tool_result` block —— 加一个**并行方法** `chatWithTools(params, tools)`,返回 `{ content: Array<TextBlock | ToolUseBlock>, stopReason }`。老 `chat()` 方法保留不动。

### 1.4 CLI 扩展

`src/cli.ts` 加 `renpy-agent v5 <inspiration> [--story <name>]` 新子命令。现有 `run` / `rebuild` / `modify` 完全不动。

### 1.5 Barrel 导出

`src/index.ts` 新增 `agents/` 导出分组,和现有 v0.2 pipeline 分组并列。

### 1.6 `src/pipeline/` 不动原则

一行不改。V5 跑起来的产物(workspace 文档、Stage A `.rpy`、AssetRegistry)走的还是串行 pipeline 建立的**同一套 JSON schema、同一个 Coder、同一个 QA** —— V5 只是换了个调度大脑,生成机器不换。这是"两个世界产物比特等价"的物理保证。

## 2. 5 个 Common Tool 的 TS 签名

### 2.1 scope 选择(W2)

v0.6 实现 **5 个**真 tool:`output_with_plan` / `output_with_finish` / `read_from_uri` / `handoff_to_agent` / `call_task_agent`。workflow 三件套(`active_workflow` / `check_workflow_params` / `get_workflow_guide`)**不实现**,接到 LLM 就返回 `{error: "v0.6 unavailable"}` stub,v0.7 再做真实现。

理由:PLAN §8 "改短发"样例全部用 5 个 common tool 就够,v0.6 Stage A demo 也不需要 workflow 引擎,收 scope。

### 2.2 Context 类型

```typescript
// src/agents/common-tools.ts

export interface CommonToolContext {
  readonly storyName: string;
  readonly gameDir: string;                   // runtime/games/<story>/game
  readonly workspaceDir: string;              // runtime/games/<story>/workspace
  readonly memoryDir: string;                 // runtime/games/<story>/planner_memories
  readonly llm: LlmClient;                    // for call_task_agent
  readonly runningHubClient?: RunningHubClient;
  readonly logger: PipelineLogger;
}
```

### 2.3 Planner 侧(3 个)

```typescript
/**
 * Planner 宣布"开始一个 task,打算这么干"。纯声明用途:
 * - 宿主把 plan 文本写进 planner_memories/<story>/log.jsonl 作为审计痕迹
 * - 回给 Planner 一个空 tool_result(确认收到),Planner 继续后续 tool_use 实际干活
 * - 不解析 plan 里的伪代码,伪代码是给人看的
 */
export interface OutputWithPlanArgs {
  readonly taskId: string;                    // e.g. "create-character-baiying"
  readonly plan: string;                      // TS 伪代码,自由文本
}
export async function output_with_plan(
  args: OutputWithPlanArgs,
  ctx: CommonToolContext,
): Promise<{ ok: true }>;

/**
 * Planner 宣布"当前 task 结束",写一条 summary 到 planner_memories。
 * 宿主在读到这个 tool_use 后:
 * 1) 追加 {taskId, summary, timestamp} 到 log.jsonl
 * 2) 向主循环返回 "task done" 信号,主循环退出当前 conversation,
 *    下一个 task 开新 conversation(天然丢掉所有 tool_use/tool_result 消息)
 */
export interface OutputWithFinishArgs {
  readonly taskId: string;
  readonly taskSummary: string;               // 1-3 句,落 planner_memories
}
export async function output_with_finish(
  args: OutputWithFinishArgs,
  ctx: CommonToolContext,
): Promise<{ ok: true }>;

/**
 * Planner 按 URI 拉一份 workspace 文档详情。
 * URI scheme = "workspace://<kind>/<slug>" (PLAN §8 的 hogi:// 本地化名)
 * 返回对应 JSON 文档原文;不存在则 tool_result 里带 error。
 */
export interface ReadFromUriArgs {
  readonly uri: string;
}
export async function read_from_uri(
  args: ReadFromUriArgs,
  ctx: CommonToolContext,
): Promise<{ kind: DocumentKind; content: WorkspaceDocument } | { error: string }>;
```

### 2.4 Executer 侧(2 个真实现 + 3 个 stub)

```typescript
/**
 * Executer 切身份:把自己变成某位 POC。宿主查 poc-registry,替换 Executer
 * 下一轮看到的 tools[] 为该 POC 的 tool-set(common + POC-specific),prompt 不变。
 * 返回值包含 toolNames 列表,让 Executer 知道自己现在能用啥。
 */
export interface HandoffToAgentArgs {
  readonly pocRole: PocRole;                  // "producer" | "writer" | "storyboarder"
                                              // | "character_designer" | "scene_designer"
                                              // | "music_director" | "voice_director"
                                              // | "sfx_designer" | "ui_designer"
                                              // | "coder" | "qa"
}
export async function handoff_to_agent(
  args: HandoffToAgentArgs,
  ctx: CommonToolContext,
): Promise<{ pocRole: PocRole; toolNames: ReadonlyArray<string> }>;

/**
 * 调用 HOGI 既有的"xx 助理"task agent。v0.6 里每个 task agent =
 * 一次单独的 LLM 调用,system prompt 写在 resources/task-agents/<name>.md,
 * 不走 Planner/Executer loop(一次性,无 tool)。
 * 已知成员(v0.6 白名单只 3 个):
 *   - character_prompt_expander        (角色生图词扩写助理)
 *   - character_main_image_generator   (wraps generateCharacterMainImage)
 *   - scene_background_generator       (wraps generateSceneBackground)
 * 其他 task agent 名字按 PLAN §8 占位,未实现返回 {error: "not implemented"}。
 */
export interface CallTaskAgentArgs {
  readonly agentName: string;
  readonly input: Record<string, unknown>;
}
export async function call_task_agent(
  args: CallTaskAgentArgs,
  ctx: CommonToolContext,
): Promise<{ agentName: string; output: Record<string, unknown> } | { error: string }>;
```

### 2.5 workflow 三件套(v0.6 stub,v0.7 实现)

```typescript
// 三件套:v0.6 全部 stub,接到就返回 {error: "v0.6 unavailable"}

export async function active_workflow(args, ctx): Promise<{ error: string }>;
export async function check_workflow_params(args, ctx): Promise<{ error: string }>;
export async function get_workflow_guide(args, ctx): Promise<{ error: string }>;
```

### 2.6 语义要点

- **`output_with_plan` 是声明性的** —— 不解析 plan 伪代码,只记录。这和 Q2 定的"纯 tool_use loop"一致。
- **`handoff_to_agent` vs `active_workflow` 语义不重叠** —— handoff 换身份(tool-set 全换),workflow 是在当前身份下临时借一套 sub-tool。v0.6 workflow stub 化,所以这个区别在 v0.7 才显现。
- **`call_task_agent` 白名单 v0.6 只 3 个** —— task agent 实际 catalog HOGI 上有十几个,v0.6 不全挪。白名单外的 agentName 直接 `{error: "not implemented"}`。

## 3. POC tool-set 结构(Tier 1 / Tier 2)

### 3.1 身份 = tool-set

每个 POC 的身份 = 一组 POC-specific tool 的 schema + 实现。Executer 统一 prompt,`handoff_to_agent("character_designer")` 之后看到的 `tools[]` 就是:

```
[...5 个 common tools, ...character_designer 专属 tools]
```

v0.6 给 **11 位 POC 全部建 `tools.ts`**,但只有接 Stage A demo 路径的才填真实现(Tier 1),其他 stub(Tier 2)。

### 3.2 Tier 1 —— v0.6 实现真 tool(Stage A 必经)

| POC | tool | 包装什么 |
| --- | --- | --- |
| 制作人 | `create_project(title, genre, tone)` | 写 `workspace/project.json`,返回 URI |
| 制作人 | `create_chapter(projectUri, outline)` | 写 `workspace/chapter.json`,返回 URI |
| 编剧 | `draft_script(chapterUri, characterUris[], sceneUris[])` | 复用 `src/pipeline/writer.ts` `runWriter`,结果存 `workspace/script.json` |
| 分镜师 | `condense_to_shots(scriptUri, max?)` | 复用 `src/pipeline/storyboarder.ts` `runStoryboarder` |
| 角色设计师 | `create_or_update_character({uri?, name, description, visualDescription, mainImageUri?})` | 写 `workspace/characters/<slug>.json`;`mainImageUri` 为 null → 维持占位 |
| 角色设计师 | `generate_character_main_image(characterUri)` | 复用 `src/executers/character-designer/generate-main-image.ts` |
| 场景设计师 | `create_or_update_scene({uri?, name, description})` | 写 `workspace/scenes/<slug>.json` |
| 场景设计师 | `generate_scene_background(sceneUri)` | 复用现有 `generateSceneBackground` |
| Ren'Py 编码师 | `write_game_project(storyboardUri, scriptUri, characterUris[], sceneUris[])` | 复用 `src/pipeline/coder.ts` `writeGameProject` |
| Ren'Py 编码师 | `swap_asset_placeholder(logicalKey, realUri)` | 复用 `src/assets/swap.ts` |
| QA | `run_qa(gameDir)` | 复用 `src/pipeline/qa.ts` `runQa` |

**合计 11 个 tool。完全是对现有函数的 tool-schema 包装,没有一个是新业务逻辑。** 这是 Q4 定的"两个世界共享同一个生成机器"的直接体现。

### 3.3 Tier 2 —— v0.6 stub 占位(Stage A demo 不必经)

| POC | stub tool | 返回 |
| --- | --- | --- |
| 分镜师 | `generate_cutscene(shotUri)` | `{error: "v0.6 not yet routed through v5"}` |
| 角色设计师 | `generate_character_expression` / `generate_character_dynamic_sprite` | 同上 |
| 场景设计师 | `generate_prop` / `generate_scene_time_variant` | 同上 |
| 音乐总监 | `generate_bgm_track(chapterUri)` | 同上 |
| 配音导演 | `generate_voice_line(scriptLineRef)` | 同上 |
| 音效设计师 | `generate_sfx(shotUri, cue)` | 同上 |
| UI 设计师 | `generate_ui_patch(screenName)` | 同上 |
| QA | `kick_back_to_coder(bugReport)` | 同上 |

这些 POC 的身份 v0.6 存在(Planner 能 `handoff_to_agent("music_director")`),但 POC 打开一看 tool 都返回"v0.7 再说",会通过 `output_with_finish` 声明"此 task v0.6 不做"。**这是故意的**:让多 POC handoff 的骨架在 v0.6 就能跑,v0.7/v0.8 只需要把 tool 里的 stub 换成真实现。

### 3.4 v0.6 Stage A demo 最小 happy path

Planner 触发的 POC 身份序列(每一个是独立 task,一次 `output_with_finish` 收尾):

```
task 1: producer        → create_project + create_chapter
task 2: character_designer × N → create_or_update_character + generate_main_image
task 3: scene_designer × M    → create_or_update_scene + generate_background
task 4: writer          → draft_script
task 5: storyboarder    → condense_to_shots
task 6: coder           → write_game_project
task 7: qa              → run_qa
```

= 7 个 task,7 次独立 conversation。每个 task 内部若干轮 tool_use。跑完产物 = Stage A playable `.rpy`,**和现有串行 pipeline 输出比特级等价**(因为 Coder / QA 走的是同一份 `coder.ts` / `qa.ts`)。

### 3.5 Workspace 持久化格式

现有 `src/pipeline/workspace.ts` 写的是三份聚合 JSON(planner / writer / storyboarder)。V5 需要**按 URI 粒度存**,每个文档一份:

```
runtime/games/<story>/workspace/
├── project.json              # 单文档
├── chapter.json              # 单文档
├── characters/
│   ├── baiying.json
│   └── kurogane.json
├── scenes/
│   ├── sakura_night.json
│   └── classroom.json
├── script.json
├── storyboard.json
└── asset-registry.json       (已存在,不动)
```

URI scheme:`workspace://character/baiying` → `workspace/characters/baiying.json`。映射在 `src/agents/workspace-index.ts` 的 `resolveUri` 里。

**两套 workspace IO 并存**:现有串行 pipeline 的 `saveStoryWorkspace` / `loadStoryWorkspace` 保持不动,V5 用一套新 workspace IO。两个世界写不同的文件 —— 可接受的冗余。v0.7 修改链条接入时再决定是否统一。

## 4. Planner / Executer loop 主循环 + "改短发" trace

### 4.1 宿主视角主循环

```typescript
// src/agents/run-v5.ts
async function runV5(params: RunV5Params): Promise<PipelineResult> {
  const ctx = buildContext(params);          // workspaceDir / memoryDir / llm / ...
  await bootstrapInspiration(ctx, params.inspiration);  // 写 workspace/inspiration.json

  while (true) {
    const task = await runPlannerTask(ctx);  // 一个 Planner conversation
    if (task.done) break;                    // Planner 调 output_with_finish 且说 "all tasks done"
  }

  // Planner loop 退出 = Stage A 完工(Coder + QA tool 都已跑过)
  return assemblePipelineResult(ctx);
}
```

### 4.2 单个 Planner task 的 conversation

```
system: PLANNER_SYSTEM_PROMPT
        + galgame-workspace.ts 全文
        + workspace index(URI/title/status/kind 列表)
        + planner_memories 全量 summaries
tools: [output_with_plan, output_with_finish, read_from_uri, handoff_to_agent]

user: "请推进项目 <story_name>。当前 workspace 状态如上。决定下一个 task 做什么。"

(多轮 tool_use ↔ tool_result ...)

assistant stop: tool_use output_with_finish(taskSummary=...)
                宿主写 memory,退出 conversation
```

**关键**:Planner 的 `tools[]` 只有 **4 个 common tool**(不含 `call_task_agent`)。它本身不调 POC-specific tool,而是 `handoff_to_agent` 启动一个 sub-conversation(Executer)。

### 4.3 Executer sub-conversation(从 Planner 的 `handoff_to_agent` 触发)

```
system: EXECUTER_SYSTEM_PROMPT (统一 7-rule)
        + 当前 POC 的 tool-set 描述
        + 相关 workspace 文档片段(按 Planner 的 handoff 意图注入)
tools: [...4 common tools except handoff_to_agent, call_task_agent, ...POC-specific tools]

user: <Planner 在 handoff 时传来的"要你干什么"context>

(多轮 tool_use ↔ tool_result ...)

assistant stop: tool_use output_with_finish(taskSummary=...)
                宿主把 taskSummary 作为 handoff 的 tool_result
                回给 Planner conversation 继续
```

**注意**:Executer conversation 也用 `output_with_finish` 收尾。它的 `taskSummary` 会变成 Planner 那边 `handoff_to_agent` tool_use 的 `tool_result` 内容。这样 Planner 记忆里只有"我让角色设计师干了 X,她说结果是 Y",没有 Executer 内部 10 轮 tool_use 的噪声 —— **这是"分层压缩"**。

### 4.4 主循环退出条件

Planner 连续两个 task 都立刻 `output_with_finish(taskSummary="no more tasks, Stage A delivered")` → 主循环退出。宿主再兜底跑一次 `runQa` 确认(防止 Planner 漏跑 QA)。

### 4.5 PLAN §8 "改短发" trace(v0.7 才实现,spec 必须写清)

用户:`renpy-agent v5 modify <story> "把白樱改成短发"`

**Planner conversation 1**(唯一 1 个 task):

```
turn 1  assistant: tool_use output_with_plan(taskId="modify-baiying-short-hair",
                    plan=`
                      await handoff_to_agent("character_designer");
                      // 角色设计师会更新 visualDescription 并级联重生主图
                    `)
turn 2  user: tool_result {ok: true}
turn 3  assistant: tool_use handoff_to_agent(pocRole="character_designer")
turn 4  user: tool_result handoff complete, toolNames=[...]
```

**Executer conversation**(character_designer 身份):

```
turn 1  user: "把 workspace://character/baiying 改成短发"
turn 2  assistant: tool_use read_from_uri(uri="workspace://character/baiying")
turn 3  user: tool_result {...当前 Character JSON...}
turn 4  assistant: tool_use create_or_update_character({
                     uri: "workspace://character/baiying",
                     visualDescription: "...短发,发梢及耳...",
                     mainImageUri: null,                      // 依赖链级联:回落占位
                   })
turn 5  user: tool_result {uri: "workspace://character/baiying", status: "placeholder"}
turn 6  assistant: tool_use call_task_agent(
                     agentName="character_prompt_expander",
                     input={characterUri: "workspace://character/baiying"})
turn 7  user: tool_result {prompt: "..."}
turn 8  assistant: tool_use generate_character_main_image(characterUri=...)
                   // Tool 内部调 runImageTask → RunningHub MJ v7 → swapAssetPlaceholder
turn 9  user: tool_result {mainImageUri: "images/characters/baiying-v2.png", status: "ready"}
turn 10 assistant: tool_use output_with_finish(taskId=...,
                     taskSummary="Regenerated baiying main image with short hair")
```

**回到 Planner conversation 1**:

```
turn 5  user: tool_result from handoff = "Regenerated baiying ..."
turn 6  assistant: tool_use output_with_finish(taskId=...,
                     taskSummary="Completed short-hair modification for baiying")
→ 主循环退出
```

**Storyboard / Script / Cutscene / .rpy 一行不改** —— 因为它们对 Character 的依赖是 URI 级,且 Coder 的 `write_game_project` 没跑(Planner 没 handoff 给 Coder),`.rpy` 保持不动。AssetRegistry 的 `baiying` 条目已从 ready → placeholder → ready(新图),下次玩家打开游戏看到的是短发白樱。

这就是 PLAN §8 / §2.4 讲的"URI 引用让改动隔离"真跑起来的样子。**v0.6 里这条路走不通**(`create_or_update_character` 是新 tool,要到 v0.7 修改链条接入时才加),但 spec 必须写清楚,否则 v0.7 的实施者不知道长什么样。

### 4.6 确认项

- **Planner 每次 task = 一个新 conversation**,`planner_memories` 通过 system prompt 全量 prepend 回来(Q3 决策)。
- **Executer 的 `output_with_finish` 的 `taskSummary` 作为 Planner 那头 `handoff_to_agent` 的 `tool_result`** —— 这是"分层压缩"的核心。

## 5. 分阶段落地(M1 → M4)

v0.6 全部实现时间预估 **~7-10 个工作日**。拆 4 个 milestone,每个都独立可验证、独立可 commit、不跑完整 smoke 也能 pass typecheck + test。

### 5.1 M1 —— LLM tool-use 底座(1-2 天)

**新增 / 修改**:
- `src/llm/types.ts`:加 `LlmToolUseBlock` / `LlmToolResultBlock` / `LlmContentBlock` 类型;`LlmChatParams` 可选 `tools?: LlmToolSchema[]`;`LlmResponse` 可选 `toolUses?: LlmToolUseBlock[]` + `stopReason`
- `src/llm/claude-client.ts`:加 `chatWithTools(params)` 方法,把 Anthropic SDK 的 `tool_use` / `tool_result` block 翻译成 `LlmClient` 中立类型
- `src/llm/claude-client.test.ts`:mock `AnthropicBedrock.messages.create`,验证 `tool_use` 路径端到端

**不碰**:现有 `chat()` 方法、现有 pipeline 一行不改。

**独立验证**:`pnpm test src/llm` 全绿 + 手工敲个 10 行脚本调 `chatWithTools` 真打一次 Bedrock(用最便宜的 echo tool),确认 tool_use / tool_result roundtrip 工作。

## 6. V5 ↔ v1.0 关系 / 测试策略 / 风险

### 6.1 V5 是 v1.0 的**严格前置**

| 特性 | 串行 pipeline 能做吗? | V5 能做吗? |
| --- | --- | --- |
| 单章节 Stage A demo | ✅ 已经能 | ✅ 能(本 spec 目标) |
| 改角色 / 改对白 / 重排 | ✅ v0.4 已做 | ✅ v0.7 接入 |
| 多章节(Chapter × N) | ❌ 串行 pipeline 是"一次性写完" —— 多章节要求"决定生成顺序、章节间引用 Chapter 2 的 Cutscene 用 Chapter 1 的角色、部分章节重跑 part",这是调度问题,写死不了 | ✅ Planner 就是调度 |
| 多路线(Route × M) | ❌ 同上,路线分支需要"要不要生成这条路线"、"这条路线的变体 Scene/Character 从哪里来"的判断 | ✅ Planner 读 `Route.conditions` 自己决定 |
| 存档点(Auto checkpoint) | ❌ 写死能放几个 checkpoint,但"哪里该放"需要读 Shot 的情绪曲线 + Script 的结构信息做推理 | ✅ QA 或 Coder POC 拿 scheduler 判断 |

**结论**:v1.0 做不了是因为没有 Planner。v0.6 把 Planner 搭起来 → v1.0 的"多章节多路线"只是在 v0.6 跑通的 7-task happy path 上**多加几个 task**(`create_chapter × N`,然后章节间 handoff)。新增代码量估计 30%(大头是 schema:`Route` / `Ending` 完整字段、存档点 `RpyFile` 扩展),**架构不动**。

**这里要明文写死**,这样 v1.0 设计时不会有人提议"绕过 V5 做个 hack 多章节"(诱惑会有:既然 v0.4 能手工分章节,为什么不再搭一层?—— 因为那会让 V5 永远写不上)。

### 6.2 测试策略(三层)

**层 1:单 tool 测试**(M2 / M3 每个 tool 都有)
- 输入合法 args → 正确副作用(workspace JSON 写入 / asset-registry 更新 / 调下游函数次数)
- 输入非法 args → tool_result 里有 error
- 都不碰 LLM,全部同步

**层 2:Planner / Executer loop 测试**(M4 `run-v5.test.ts`)
- Scripted LLM:mock `chatWithTools` 按预定 tool_use 序列回应。例如:
  ```
  Planner turn 1 → tool_use handoff_to_agent("producer")
  Executer turn 1 → tool_use create_project(...)
  Executer turn 2 → tool_use create_chapter(...)
  Executer turn 3 → tool_use output_with_finish(...)
  Planner turn 2 → tool_use handoff_to_agent("character_designer")
  ...
  ```
- 验证:产物与"直接跑串行 pipeline + 同一个 inspiration"**比特级等价**(因为 Coder / QA 一样)
- 这个测试是 V5 的护城河:确保 loop 改来改去不打破最基本 happy path

**层 3:真 key smoke**(M4 交付标准)
- 模仿 v0.5 `scripts/modify-smoke.mjs` 的结构,做 `scripts/v5-smoke.mjs`(可以 v0.7 再做)
- v0.6 交付一次手工 smoke 足矣,结果粘 `PLAN.md` v0.6 条目

**不做**:单独 mock 整个 Bedrock 的集成测试。Scripted LLM 已经覆盖控制流,真 key smoke 覆盖 end-to-end,中间层多做一遍不增价值。

### 6.3 风险清单

| 风险 | 可能性 | 缓解 |
| --- | --- | --- |
| Planner tool-use 发出的 args 不满足 Tier 1 tool 的 schema 要求(JSON Schema validation fail) | 高 | 每个 tool 入口做 zod / 手写 assertion,失败返回清晰 tool_result error,让 Planner 自修 |
| Planner 死循环(重复 handoff / 永不 `output_with_finish`) | 中 | 每个 conversation 设 `max_turns = 30`,超时强制 `output_with_finish` + 日志 |
| Executer 在没先 `handoff_to_agent` 时就调 POC tool | 低 | Executer system prompt 强调 "rule 1: must handoff first";tool dispatch 时校验当前身份,非法调用返回 error |
| Bedrock 偶发 tool_use block 带 invalid JSON(已知 Sonnet 4.6 偶发) | 中 | tool parse 失败 → tool_result 带 error,让模型重试;超 3 次重试 → 整 task fail |
| Planner system prompt 超 context(一次灌 schema + index + memories) | 低 | v0.6 单项目规模小,Sonnet 4.6 200k 够用;到量大时 memory prepend 改成 top-N 或 token-budget |
| URI 冲突(两个角色都叫"白樱") | 中 | `workspace-index` 在 `create_or_update_character` 时对 slug 做去重,同 slug 报错让 Planner 改名 |
| v5 产物和串行 pipeline 产物不比特等价(测试盲点) | 中 | M4 `run-v5.test.ts` 专门 diff 两路产物;发现差异优先收敛到共同的 Coder 输出 |
